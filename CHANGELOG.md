### Changed
- **LLDP polling moved to direct SNMP (Block C P5).** `polling.collect_lldp` now walks `lldpRemTable` + `lldpRemManAddrTable` via `lldp_snmp.collect_lldp` instead of routing through NAPALM proxy. Group-by-local-interface adapter feeds the existing `upsert_node_lldp_bulk` NAPALM-shape contract; `snmp_collector.collect_ifindex_to_name` resolves each neighbor's `local_port_ifindex` to a Junos-style local interface name (sentinel `ifindex:N` on miss preserves the entry per Rule 7). **First populator of the 5 expansion columns added by P2 migration `c3208527926f`** — `local_port_ifindex`, `local_port_name`, `remote_chassis_id_subtype`, `remote_port_id_subtype`, `remote_system_description`. `upsert_node_lldp_bulk` extended: writes the 5 new columns from per-neighbor `n.get(...)` with `None` default (NAPALM-shaped callers omit them and write NULL), and `on_conflict_do_update` set_ block refreshes them alongside the legacy columns so re-polls track changes when neighbors flip subtypes or sysDescr. **Adapter dedupes by `(local_iface, remote_system_name_or_empty, remote_port_or_empty)`** matching the `uq_lldp_node_iface_remote` unique constraint — observed live on EX4300-48T `ge-0/0/24` where two unmanaged neighbors advertise distinct chassis IDs but the same `port-MAC` identifier and `sys_name=None`; without dedup PostgreSQL `ON CONFLICT DO UPDATE` raises `CardinalityViolationError` and the BLE001 guard silently drops the entire batch (entire device's neighbors lost). Same class of bug as P3's Junos `lo0.X` collision; same fix discipline (first-seen wins). NAPALM-path `_collect_lldp_napalm_deprecated` retained as an unreferenced function pending Block C P6 deletion. New event keys: `lldp_snmp_collect_start` (DEBUG), `lldp_snmp_collect_complete` (INFO; includes `neighbors_raw` / `neighbors_persisted` / `local_interfaces` / `sentinel_neighbors` counts), `lldp_snmp_collect_failed` (WARN). 14 new unit tests cover happy path with 5 expansion fields populated, sentinel group key on ifIndex resolution miss, collector-pre-resolved `local_port_name` preservation, `lldp_snmp.collect_lldp` SnmpTimeoutError → row failure, empty name map → all-sentinels graceful degradation, empty neighbor list, multiple neighbors per local interface (LLDP allows it), no-IP fail-fast, credential hygiene, status-gate non-duplication, dedup discipline (constraint-key collision + distinct-port non-collision), P6 deletion-target guard. Live-validated against 5 lab devices: EX2300/EX3300/EX4300/SRX320 all populate the 5 expansion columns at 100% fill rate (chassis_id_subtype always `mac_address` on Junos; port_id_subtype is `interface_name` for managed neighbors and `mac_address` for unmanaged neighbors like Mist APs and shared-segment endpoints; remote_system_description populated where the neighbor advertises it — typical 70-100% per-device); PA-440 returns 0 neighbors (legitimate — firewall does not run LLDP). FG-40F + arista-mnm `lldp` rows remain operationally disabled pending P6 reversal.
- **MAC polling moved to direct SNMP (Block C P4).** `polling.collect_mac` now walks `dot1qTpFdbTable` / `dot1dTpFdbTable` via `mac_snmp.collect_mac` instead of routing through NAPALM proxy. Bridge-port resolution chain: `mac_snmp.collect_mac` returns `MacEntry.bridge_port` (raw `dot1dBasePort` integer); the new `snmp_collector.collect_bridgeport_to_ifindex` walker (BRIDGE-MIB `dot1dBasePortIfIndex`, OID 1.3.6.1.2.1.17.1.4.1.2) maps bridge port → ifIndex; existing `snmp_collector.collect_ifindex_to_name` then maps ifIndex → interface name (e.g. `ge-0/0/12`). On bridge-port resolution miss the sentinel uses the bridge_port value (`ifindex:<bridge_port>` — operators can correlate back to the FDB row); on ifIndex name miss the sentinel uses the resolved ifIndex (`ifindex:<ifIndex>`). Either failure mode preserves the entry per Rule 7 — the row lands with a sentinel interface name, the poll cycle is still SUCCESS. Sentinel classifier behaviour pinned in tests: `_is_access_interface("ifindex:N")` returns True (none of `_NON_ACCESS_PREFIXES` match), `_infer_vlan_from_interface("ifindex:N")` returns 0 (regex doesn't match) — the operator-visible result is "MAC on access port, VLAN 0". `entry_status` (`learned` / `self` / `mgmt` / `other` / `invalid` from MacEntry) maps to the upsert's `static` bool: `self` and `mgmt` → `static=True`, everything else → `static=False`. Adapter dedupes by `(MAC.upper(), interface, vlan_int)` matching the `uq_mac_node_mac_iface_vlan` unique constraint — defensive against orphan-FDB rows that all coerce to vlan=0 (Junos cases the standard VLAN map can't resolve). NAPALM-path `_collect_mac_napalm_deprecated` retained as an unreferenced function pending Block C P6 deletion after a 1-cycle soak window. New event keys: `mac_snmp_collect_start` (DEBUG), `mac_snmp_collect_complete` (INFO; includes `entries` and `sentinel_entries` counts), `mac_snmp_collect_failed` (WARN). 19 new unit tests cover happy path, bridge-port miss → bridgeport sentinel, ifIndex miss → ifindex sentinel, both maps empty (graceful degradation), `mac_snmp.collect_mac` SnmpTimeoutError → row failure, VLAN None→0 coercion + real-VLAN passthrough, entry_status remap (parametrized over learned/self/mgmt), sentinel classifier behaviour pin, empty MAC table, no-IP fail-fast, credential hygiene, status-gate non-duplication, dedup discipline including case-insensitive MAC handling, and P6 deletion-target guard. Live-validated against 4 Junos lab devices (EX2300/EX3300/EX4300/SRX320 — FG-40F + arista-mnm rows remain operationally disabled pending P6 reversal): all four poll cycles complete in 1.2-5 seconds with `last_error=null`, real interface names (`ae0`, `ge-0/0/12`, etc.) resolve correctly, real 802.1Q VLAN IDs (10, 51, 52, 110, 130, 140, etc.) populated via the existing Junos `jnxL2aldVlanTable` fallback, sentinel rate is 0 on Junos switches that fully implement BRIDGE-MIB and 2 entries on EX3300 (legitimate `bridge_port=0` system entries — not a bug). PA-440 returns 0 entries (legitimate — firewalls don't populate FDB tables outside L2 zones). SNMP returns more entries per cycle than NAPALM did, consistent with the Phase 2.7 OctetString-bug lesson.

### Fixed
- **Orchestrator Step D missing `parent`/`namespace`.** Nautobot 3.x rejects IP creation with `400: "One of parent or namespace must be provided"`. The orchestrator now captures the covering prefix UUID from Step C and passes it as `parent_prefix_id` to `create_ip_address`. Caught during vEOS live validation.
- **Orchestrator IPAM pre-clean before Step D.** MNM's sweep pipeline records every alive IP into Nautobot IPAM, so by the time a device is onboarded, its management IP is typically already a standalone IPAM record. The orchestrator now calls `delete_standalone_ip(ip)` before Step D (same pattern the legacy plugin path uses — see Phase 2.5 "IPAM collision on onboarding" lesson). Strict-new pre-check at Step 0.5 guarantees this pre-clean is safe.
- **Strict-new pre-check graceful failure on bad input.** Bogus `location_id` caused `find_device_at_location` to raise 400; the orchestrator now returns a clean `OnboardingResult` pointing at the likely culprit instead of propagating the exception.
- **`onboard_probe.py` CLI now initialises the controller DB.** `ensure_device_polls` silently no-ops when the DB module isn't ready — that's by-design for degraded startup, but standalone `python -m` invocation runs outside the FastAPI lifespan. The CLI now calls `db.init_db()` before dispatching so polling rows actually get seeded.
- **`delete_standalone_ip` assignment-check hardening.** Was reading the `interfaces` attribute from the IP record with `depth=0` and interpreting a null/empty list as "no assignment" — which let it delete primary IPs whose link lives in the through-model row. Caught during Prompt 6 live validation when a buggy pre-clean on `arista-mnm`'s primary IP silently orphaned the device's `primary_ip4`. Now queries `/api/ipam/ip-address-to-interface/?ip_address=<uuid>` explicitly; if any link row exists the IP is left alone.
- **`get_device` accepts `depth=` kwarg.** Required for callers (Phase 2) that need the nested `primary_ip4.address` string. Default behaviour (`depth=0`) unchanged.
- **(D2)** Removed duplicate `POST /api/discover/onboard` route registration in `controller/app/main.py`. The earlier handler (sweep-schedule-retry convenience path; only `ip` field) was dead code — the UI routes through `/api/onboarding/direct-rest` since Prompt 8, and the documented API contract (`docs/API.md`) matches the surviving 3-field handler.
