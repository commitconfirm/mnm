# v0.9.0 Installation Issues

Issues discovered during clean-install validation on a fresh Ubuntu 24.04 VM (mnm-test), running v0.9.0 tag, April 2026. These become input for v1.0 stability work.

---

### Issue: Controller starts before bootstrap creates its database

**Severity:** Blocker — **RESOLVED in v0.9.1** (see `fix/controller-db-race` branch)

**Discovered during:** Phase 3 / Phase 4

**Symptom:** After `docker compose up -d`, the controller reports `db_connected: false` in `/api/health`. Controller logs show: `database "mnm_controller" does not exist — falling back to JSON`. If you don't notice this and don't restart the controller, polling, endpoint correlation, and sweep history are all silently broken — data goes into an in-memory JSON store that disappears on restart.

**Workaround applied:** Run `docker restart mnm-controller` after bootstrap completes. The controller reconnects to the now-existing database on startup. Documented in DEPLOYMENT.md as an explicit required step.

**Root cause guess:** Bootstrap creates the `mnm_controller` database as part of its run. The controller's `init_db()` is called at container startup, before bootstrap runs. There's no retry loop — it attempts once, fails, logs a warning, and continues in JSON fallback mode.

**Suggested fix:** Add a retry loop to `init_db()` (e.g., attempt every 30 seconds for 5 minutes) so the controller recovers automatically once bootstrap creates the database. Alternatively, bootstrap could be baked into the container entrypoint so it runs before the API server starts. The README's quickstart should explicitly say "restart the controller after bootstrap" until the retry loop exists.

### Issue: "Failed to get Nautobot API token" on fresh deploy

**Severity:** Major

**Discovered during:** Phase 3 (controller startup logs)

**Symptom:** On first startup, before bootstrap runs, the controller logs multiple `WARNING: Failed to get Nautobot API token` events. As a result: poll seeding fails silently (no devices seeded to device_polls), the node info warmup loop fails, and SNMP target discovery fails. These features all depend on having a valid Nautobot API token for the `mnm-admin` user — which doesn't exist until bootstrap creates it.

**Workaround applied:** Restarting the controller after bootstrap resolves this — bootstrap creates the user and token, controller fetches it on startup. See the blocker issue above.

**Root cause guess:** The controller uses a lazy token-fetch via `docker exec` into the Nautobot container. On cold boot, the Nautobot superuser doesn't exist yet (bootstrap hasn't run), so the exec fails. There's no retry mechanism.

**Suggested fix:** Same as the database issue — a retry loop on token acquisition so the controller recovers once bootstrap runs without needing a manual restart. The retry loop and DB retry could share the same recovery path.

### Issue: `.env.example` claims passwords are "auto-generated if left blank"

**Severity:** Major

**Discovered during:** Phase 2 (reading .env.example before creating .env)

**Symptom:** The `.env.example` header says "All passwords will be auto-generated if left blank." This is false for at least two critical variables:

- `MNM_ADMIN_PASSWORD`: bootstrap.sh uses `${MNM_ADMIN_PASSWORD:?MNM_ADMIN_PASSWORD must be set in .env}` which immediately exits with a fatal error if blank
- `NAUTOBOT_SECRET_KEY`: Django requires a non-empty SECRET_KEY. A blank value would likely cause Nautobot to fail to start with an ImproperlyConfigured error (untested, but standard Django behavior)

**Workaround applied:** Set all passwords explicitly in `.env` before running bootstrap. Documented in DEPLOYMENT.md.

**Root cause guess:** The "auto-generated" comment is aspirational (perhaps from a planned feature) or was accurate for an older version.

**Suggested fix:** Remove the "auto-generated if left blank" comment or implement actual auto-generation (generate random passwords, write them back to `.env`, print them to console). The minimal fix is to update the comment to say which fields are required vs optional. The DEPLOYMENT.md takes the honest approach: mark `MNM_ADMIN_PASSWORD` and `NAUTOBOT_SECRET_KEY` as required.

### Issue: Bootstrap "Next Steps" output directs operator to wrong UI

**Severity:** Minor

**Discovered during:** Phase 4 (reading bootstrap completion output)

**Symptom:** Bootstrap prints:

```
Next steps:
  1. Log in to Nautobot
  2. Go to Jobs > Sync Devices From Network
  3. Enter a device IP, select 'Default Site'
  4. Create credentials in Secrets, then run the job
```

This is the Nautobot-native workflow for the device onboarding plugin. MNM's actual operator workflow is to use the controller's Discovery page (`http://<host>:9090`), not navigate Nautobot's job interface.

**Workaround applied:** Ignored the bootstrap output, used the controller UI.

**Root cause guess:** Bootstrap text was written early in the project before the controller's Discovery page became the primary interface. It was never updated.

**Suggested fix:** Update bootstrap completion output to say: "Open the MNM controller at http://localhost:9090 → Operations → Discovery → add a CIDR range and credentials → Start Sweep."

### Issue: Sweep API field names not documented and differ from UI form

**Severity:** Minor

**Discovered during:** Phase 5 (first sweep attempt via API)

**Symptom:** First sweep attempt used `{"cidrs": [...]}` (reasonable guess from looking at the UI). The API returned a validation error:

```json
{"detail": [{"type": "missing", "loc": ["body", "cidr_ranges"], ...},
            {"type": "missing", "loc": ["body", "location_id"], ...},
            {"type": "missing", "loc": ["body", "secrets_group_id"], ...}]}
```

Correct field names are `cidr_ranges` (not `cidrs`), and both `location_id` (a Nautobot UUID) and `secrets_group_id` are required by the schema. Getting `location_id` requires a separate Nautobot API call. There's no documentation of this in API.md or the README.

**Workaround applied:** Checked `/openapi.json` for the actual schema. Fetched `location_id` from Nautobot API. Passed `secrets_group_id: ""` (empty string) for a sweep-only run with no onboarding.

**Root cause guess:** The OpenAPI schema is correct — FastAPI generates it automatically. The documentation gap is that API.md doesn't document the sweep endpoint's parameters or the pattern for getting a `location_id`.

**Suggested fix:** Add the sweep endpoint to API.md with a complete curl example showing how to get the `location_id` from Nautobot first, and note that `secrets_group_id` can be empty string for sweep-only (no onboarding) runs.

### Issue: `mnm-gnmic` and `mnm-traefik` have no Docker healthchecks

**Severity:** Cosmetic

**Discovered during:** Phase 3 (reading `docker compose ps` output)

**Symptom:** `docker compose ps` shows these two containers as `Up` without `(healthy)`. The controller's `/api/health` endpoint reports `containers_healthy: 9` out of 11, which looks alarming but is actually the expected state. An operator who doesn't know this will investigate a non-problem.

**Workaround applied:** Explained in DEPLOYMENT.md that this is expected.

**Root cause guess:** gnmic and Traefik were added without healthchecks. Traefik's healthcheck would require a Traefik-specific health endpoint. gnmic's healthcheck could ping its Prometheus metrics endpoint.

**Suggested fix:** Add `healthcheck` blocks to both services in `docker-compose.yml`. Traefik: `test: ["CMD", "wget", "--quiet", "--tries=1", "--spider", "http://localhost:8080/ping"]` (Traefik has a built-in `/ping` endpoint when the `ping` entrypoint is enabled). gnmic: check its Prometheus metrics port. This would make `containers_healthy: 11/11` the expected state.

### Issue: `/api/health` consistently takes 1–2 seconds and logs `slow_request` warnings

**Severity:** Minor

**Discovered during:** Phase 4 (controller log review)

**Symptom:** Every call to `/api/health` takes 1,000–1,600ms and triggers a `slow_request` WARNING in the structured log. Since the Docker healthcheck calls this endpoint every ~15 seconds, the controller log is flooded with slow_request warnings, making it hard to spot real errors.

**Workaround applied:** None needed for functionality. Noted in DEPLOYMENT.md troubleshooting section.

**Root cause guess:** `/api/health` likely calls multiple downstream services synchronously (Nautobot API, Docker SDK, PostgreSQL) on every request. Each is fast, but the sum crosses the slow-request threshold. Either the threshold is too low for an aggregating endpoint, or the health check does too much work on each call.

**Suggested fix:** Two options: (a) cache the health response for 10–30 seconds and return stale data to the healthcheck — Docker healthchecks just need a non-500, not real-time data; (b) raise the `slow_request` threshold specifically for `/api/health`, or exclude it from slow_request logging entirely.

### Issue: First-boot time not documented anywhere

**Severity:** Minor

**Discovered during:** Phase 3 (waiting for containers)

**Symptom:** An operator running MNM for the first time has no indication that the full startup sequence takes ~16 minutes: 6–7 minutes for all containers to become healthy, then ~9 minutes for bootstrap to complete. Nautobot alone takes 4–5 minutes on first boot while Django runs database migrations. Without documentation, operators commonly assume something is broken and try to restart containers.

**Workaround applied:** Documented timing in DEPLOYMENT.md.

**Root cause guess:** Not a bug — just missing documentation.

**Suggested fix:** The README quickstart should include a note like "Allow 15–20 minutes for first boot. Nautobot takes several minutes to initialize. Bootstrap takes ~10 minutes to import the device type library. Both are one-time costs." Optionally, the bootstrap script could display an estimated time remaining during the device type import phase.

### Issue: Bootstrap output includes Nautobot internal URL, not external port

**Severity:** Cosmetic

**Discovered during:** Phase 4 (reading bootstrap output)

**Symptom:** Bootstrap completion prints:

```
Nautobot URL:  http://localhost:8443
```

This is correct for host-local access, but an operator accessing MNM remotely will use a different hostname. The URL printed is useful, but only if you're on the MNM host itself.

**Workaround applied:** None needed.

**Root cause guess:** Bootstrap hardcodes `localhost` as the hostname. It doesn't know the external hostname or IP.

**Suggested fix:** Use `$MNM_DOMAIN` from the `.env` if set, falling back to `localhost`. Print both the controller URL and Nautobot URL: "Controller: http://<domain>:9090, Nautobot: http://<domain>:8443".
